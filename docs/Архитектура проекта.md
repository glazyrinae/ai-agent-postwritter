# Архитектура проекта

## Общая идея

Проект построен как Docker-first сервис для работы с базовой моделью через `vLLM`, LoRA-адаптерами и HTTP API поверх них.
Архитектура намеренно разделена на несколько простых слоев, чтобы runtime-код не смешивал entrypoint, бизнес-логику и интеграции.

## Слои

- `src/app/` - создание FastAPI приложения, lifecycle, настройки и сборка зависимостей.
- `src/core/` - общие инфраструктурные элементы: логирование и единый формат ошибок.
- `src/features/agents/` - низкоуровневые операции с агентами и пайплайнами.
- `src/features/articles/` - orchestration длинной статьи: outline, главы, summary-контекст, conclusion, финальная сборка markdown.
- `src/integrations/vllm_server/` - клиент к OpenAI-compatible API сервера `vLLM`.
- `src/integrations/langchain/` - LangChain orchestration, которое собирает prompt-цепочки и вызывает agent aliases через `vLLM`.
- `deploy/models/` - локально смонтированная базовая модель для `model-downloader` и `vllm-server`.
- `deploy/lora_adapters/` - локально смонтированные LoRA-адаптеры для `vllm-server` и API.
- `deploy/gguf_adapters/` - GGUF-адаптеры для Ollama, сконвертированные из LoRA.
- `deploy/modelfiles/` - базовый fallback `Modelfile` и сгенерированные `Modelfile` для перерегистрации Ollama-моделей.

## Границы ответственности

- `app` ничего не знает о prompt engineering и не содержит бизнес-логики генерации.
- `features/agents` и `features/articles` не должны вручную оркестрировать agent steps прямыми вызовами к `OpenAI`; orchestration идет через LangChain abstraction.
- `integrations` не принимают продуктовых решений и не управляют flow статьи.
- `core/errors` отвечает за стабильный формат ошибок API.

## Pipeline-слой

`POST /pipeline` используется как LangChain pipeline поверх agent aliases в `vLLM`.

1. Клиент передает исходный текст и список alias, например `["summarizer", "writer"]`.
2. LangChain собирает последовательность шагов.
3. Каждый шаг вызывает отдельный agent alias в `vLLM`.
4. Финальный ответ последнего шага возвращается клиенту.

## Генерация длинной статьи

Фича статей работает синхронно в рамках одного HTTP запроса:

1. Получить или сгенерировать outline.
2. Распарсить outline в секции.
3. Последовательно генерировать главы.
4. После каждой главы через отдельный агент `summarizer` сгенерировать краткое summary.
5. Использовать summary предыдущих глав как контекст для следующей.
6. Отдельно сгенерировать заключение.
7. Собрать финальную статью в Markdown.

## Что пока не входит

- База данных.
- Очереди и фоновые job’ы.
- Хранение результатов генерации.
- Отдельный CLI для article generation.
